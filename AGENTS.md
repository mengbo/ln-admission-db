## 📖 术语约定

本项目把以下说法视为**同义词**，指代同一个产出物：

| 同义说法 | 说明 |
|---|---|
| **志愿填报方案**（正式名） | AI 一次性按 [templates/report.md](../templates/report.md) 输出的完整文档 |
| 志愿填报报告 | 历史叫法，与"方案"等价 |
| 志愿填报指南 | 偏向"可参考的执行手册"语感，与"方案"等价 |
| 志愿方案 | 简称，与"方案"等价 |
| 报告 / 方案 / 指南 | 单字简称，默认指志愿填报方案 |

**AI 行为要求**：

1. **正式输出物的标题**统一用 **"志愿填报方案"**——例如方案头第 1 行写 `# 志愿填报方案`
2. **正文行文中**可以混用"报告/指南/方案"等说法，AI 理解即可，**不必机械替换**——保证行文自然
3. **当用户说"给我一份报告/方案/指南"时**，直接按 [templates/report.md](../templates/report.md) 输出，不要因为用词不同而追问"您是要报告还是方案"
4. **存档目录**保持 `reports/` 不变（已在 `.gitignore` 中，本地约定）；文件名加时间戳，详见「📁 方案存档约定」
5. **文档文件名 `report.md` 保持不变**——历史原因，改名涉及所有引用链接；正文标题改即可

## 构建/设置命令
- `./scripts/createdb.sh` - 创建 SQLite 数据库并导入所有 CSV 数据
- `./scripts/exp_phy2024.sh` - 运行 2024 年物理类数据示例查询
- `sqlite3 data.sqlite3` - 直接打开数据库进行查询
- `sqlite3 -cmd ".mode csv" -cmd ".headers on" data.sqlite3 < queries/query.sql` - 运行 SQL 文件并以 CSV 格式输出

## 查询方式约定

**本项目直接使用 `sqlite3` 命令行工具进行数据查询，不使用 MCP sqlite 工具。**

理由：
- `sqlite3` 命令行与现有 shell 脚本（如 `scripts/exp_phy2024.sh`）风格一致
- SQL 语句可保存为文件复用、便于版本管理
- 无 MCP 协议开销，响应更快
- 任何能执行 shell 的环境都能用

常用查询模板：

```bash
# 列出所有表
sqlite3 data.sqlite3 ".tables"

# 查看表结构
sqlite3 data.sqlite3 ".schema table_name"

# 表格对齐输出（带表头）
sqlite3 -header -column data.sqlite3 "SELECT * FROM table_name LIMIT 5;"

# CSV 格式输出
sqlite3 -header -csv data.sqlite3 "SELECT score, rank FROM rank_phy2026 WHERE CAST(score AS INTEGER) = 424;"

# 一行内联查询
sqlite3 data.sqlite3 "SELECT COUNT(*) FROM plan_2026;"

# 复杂查询保存为 SQL 文件
sqlite3 -header -column data.sqlite3 < query.sql
```

## 代码风格指南
- **重要：所有交互过程、注释和文档必须使用中文，本项目为中文项目**
- **优先使用 `sqlite3` 命令行进行数据查询，不使用 MCP sqlite 工具**（如需 MCP 方案请参考 [docs/MCP.md](./docs/MCP.md)）
- SQL 查询应使用正确的 JOIN 语法和表别名（如 `p`, `s`, `r`）
- 遵循现有表命名规范：`plan_YYYY`, `rank_hisYYYY`, `rank_phyYYYY`, `score_hisYYYY`, `score_phyYYYY`
- 使用描述性列名匹配模式：`inst_code`, `major_code`, `min_score`, `plan_num`
- Shell 脚本应可执行并使用正确的 shebang `#!/bin/bash`
- CSV 导入应保持一致的数据类型（数字用 INTEGER，字符串用 TEXT）
- 使用 SQLite 特定语法如 `||` 进行字符串连接，`LIKE` 进行模式匹配
- Shell 脚本中包含适当的错误处理，变量用引号包围
- Shell 脚本中的多行 SQL 查询使用 heredoc（`<<EOF`）
- SQL 格式化应使用适当的缩进和一致的列顺序
- 数值范围使用 `BETWEEN`，模式匹配使用 `LIKE` 与 `%`
- 复杂的 WHERE 子句和过滤逻辑应包含注释（`--`）

## 📌 辽宁志愿模式（核心概念）

> **先把这个搞清楚，再看其他章节。** 这是整个项目分析方法的根基。

辽宁采用 **"专业平行志愿"** 模式，与其他省份的"院校+专业组"模式有本质区别：

| 维度 | 辽宁专业平行志愿 | 其他省"院校+专业组" |
|---|---|---|
| 1 个志愿 = ？ | 1 个 **(院校, 专业)** 组合 | 1 个 (院校, 专业组)，组内 4-6 个专业 |
| 志愿容量 | **112 个** | 通常 45 个左右 |
| 是否调剂 | **不调剂** | 服从调剂可能被调剂到组内其他专业 |
| 投档方式 | 分数优先、遵循志愿、一次投档 | 同 |
| 专业要求 | 须满足该专业的 `first` + `second` | 须满足该专业组的要求 |

**对本项目的关键影响**：

1. **数据库的每行 ≈ 1 个辽宁志愿单位**
   - `plan_YYYY` / `score_phyYYYY` / `score_hisYYYY` 的每一行 = 1 个 (院校, 专业) 组合
   - 也就是 1 个可填的辽宁志愿槽
2. **"按 (院校, 专业) 去重"的真正含义**
   - 不是 SQL 层面的数据去重
   - 而是 **志愿单位层面的去重**：同一个 (院校, 专业) 不管在候选池中出现几次，最终志愿表只占 1 槽
3. **选科一致性必须每次检查**
   - 辽宁模式下，如果考生选科不满足专业的 `first`/`second` 要求，该志愿直接无效
   - 志愿筛选时务必带 `p.first LIKE ... AND p.second LIKE ...` 过滤
   - 选科结构详细说明见下一节「📚 选考科目结构（核心概念）」

## 📚 选考科目结构（核心概念）

> 这是辽宁"3+1+2"新高考选科模式下的术语，对应数据库 `plan_YYYY.first` 和 `plan_YYYY.second` 两个字段。

### 两个维度

- **首选科目（`plan_YYYY.first`）**：**物理 / 历史** 二选一（这是"3+1+2"里的那个"1"）
  - 决定考生走"物理类"还是"历史类"赛道
  - 物理类 → `p.first LIKE '%物%'`
  - 历史类 → `p.first LIKE '%历%'`
- **次选科目（`plan_YYYY.second`）**：从化学、生物、政治、地理中选 1-3 门（"3+1+2"里的那两个"2"）
  - **多数专业的 `second` 是"不限"**——这是默认情况，AI 不必为次选科目过度筛选
  - 少数专业有具体要求，常见：
    - `化学`（化学必选，医、工、师范高频）
    - `化学或生物`（化学、生物二选一）
    - `政治`（部分文科类）
    - `生物`、`地理` 等

### 关键事实

- **首选不满足**：该志愿**完全无效**（物理类看历史类志愿是白看）
- **次选不满足**：该志愿**同样无效**（专业要求化学但考生没选，直接淘汰）
- **"化学或生物"型要求按"任一满足"处理**——不要误判为"两门都必选"
- **志愿筛选时务必带 `p.first LIKE ... AND p.second LIKE ...` 过滤**

### 对 AI 出方案的隐含指引

- **不要为次选科目过度筛选**：多数志愿 `second = '不限'`，默认就放行
- **真正需要注意的是"次选有具体要求"的那批专业**——这些是真正的"门槛志愿"
- 物理类 + 化学/生物 选科 → 视野广（医、工、师范、"不限"类都可填）
- 历史类 + 政治/地理 选科 → 视野比物理类窄，但仍有大量"不限"可选
- **选科不一致的志愿必须从候选池中剔除**，否则最终志愿表会出现"投档了却没资格"的情况

## 📅 年份约定（通用术语）

为了让方案与具体年份解耦，本项目所有分析都使用相对概念：

- **今年** = 考生参加高考的年份（AI 从考生信息中获取）
- **去年** = 今年 − 1
- **前年** = 今年 − 2

SQL 查询时按"今年"的具体年份替换 `YYYY`：

| 今年 | 数据表使用 |
|---|---|
| 2026 | `rank_phy2026`（今年）+ `rank_phy2025`（去年）+ `rank_phy2024`（前年） |
| 2027 | `rank_phy2027`（今年）+ `rank_phy2026`（去年）+ `rank_phy2025`（前年） |
| 2028 | `rank_phy2028`（今年）+ `rank_phy2027`（去年）+ `rank_phy2026`（前年） |

**项目维护**：每年补充新一年的 CSV 数据后，所有模板和 SQL 都不需要改框架，只需要把"今年"重新指向新一年即可。

## 📋 当年数据 vs 历史数据（核心原则）

> **方案默认服务于考生当年的高考志愿填报**——这是 AI 出方案时必须分清的第一原则。

| 数据类型 | 使用原则 | 理由 |
|---|---|---|
| **招生计划（`plan_今年`）** | **必须用当年**，往年仅参考 | 今年招 15 人就是 15 人；去年招 30 不代表今年还招 30 |
| **录取分数（`score_去年`/`score_前年`）** | 历史数据，用于志愿分析 | 等效分换算、录取格局判断都依赖历史 |
| **一分一段表（`rank_今年`/`rank_去年`/`rank_前年`）** | 历史数据，用于等效分换算 | 位次是沟通今年与历史的稳定坐标轴 |

### 关键约束

- **方案中的"计划数"列必须取自 `plan_今年`**，写 SQL 时明确 `FROM plan_今年 p`
- 往年的招生计划数（`plan_去年`/`plan_前年`）仅作风险章节的背景参考，**不能代入志愿表**
- 风险章节可以提"该校该专业历年计划数变化趋势"，但**不能据此调整今年的招生计划数**
- 如果 `plan_今年` 缺数据（极少见），提醒家长核对官方最新发布；**不要用去年数据填空**

### 反例

❌ "去年招 30，今年应该也差不多吧" → 用去年 30 人填今年计划数
❌ "前年招 20、去年招 25，今年招 15 应该是波动" → 把今年 15 改成 25

✅ 今年 `plan_2026` 显示 15，方案里就写 15；备注里可写"去年招 25、前年招 20，今年缩减至 15"作为风险提示

## 📝 院校简介：依赖模型知识，不保证正确（核心原则）

> **方案中"院校简要介绍"一节由 AI 大模型根据训练知识生成——可能不准确、不完整、有时效性问题**。
> 这是本项目**唯一不来自数据库**的内容，必须明确告知读者。

### AI 行为要求

- 院校简介内容**仅依赖 AI 模型自身知识**，不强制要求联网搜索
- **如模型对该院校无相关知识**：必须明确写"院校信息不详，请考生自行了解"——**不要瞎编、不要泛泛而谈、不要把相近学校的信息套上去**
- 每条简介尽量简短（不超过 50 字），**宁可少写、不可错写**
- **降级策略**：模型对某院校完全无知识时，直接在表里该行写"院校信息不详，请考生通过学校官网 / 教育部官网 / 阳光高考平台了解"，不要硬填

### 报告中的必备声明

[templates/report.md 第 5 节](../templates/report.md#5-院校简要介绍) 开头必须有一段醒目的免责声明：

> 本节内容由 AI 大模型根据其训练知识生成，**不保证准确无误，仅供参考**。
> 如院校信息不详或与官方不一致，请以学校官网、教育部官网、阳光高考平台等权威来源为准，**考生务必自行核实后再做选择**。

### 反例

❌ 模型不知道某院校 → 编造一段看似合理但虚构的校史
❌ 模型对该院校仅有模糊印象 → 写得含含糊糊但又不注明"信息不详"
❌ 用其他学校的特色专业"嫁接"过来填补空白

✅ 模型不知道某院校 → 表里该行直接写"院校信息不详，请考生自行了解"
✅ 模型只知道片段 → 把已知信息写出来，再标注"以上为模型已知信息，建议考生进一步核实"

## 🤖 AI Agent 辅助报考工作流

### 推荐工具

- **[OpenCode](https://opencode.ai)** / **Claude Code**：本地 shell 即可调 `sqlite3`，零额外配置，响应快。
- 不推荐：Cline / Cherry Studio 通过 MCP 协议接入，仅作历史参考，详见 [docs/MCP.md](./docs/MCP.md)。

> 推荐理由：Agent 拥有 shell 执行权，可以直接运行 `sqlite3` 命令；省去 MCP 协议握手、SQL 写入与回读的开销，更稳定。

### Agent 调 sqlite3 的最佳实践

1. **可读输出**：用 `-header -column` 让 Agent 更易解析。
   ```bash
   sqlite3 -header -column data.sqlite3 "SELECT ..."
   ```
2. **多行 SQL 用 heredoc**：避免 shell 转义地狱。
   ```bash
   sqlite3 -header -column data.sqlite3 <<'EOF'
   SELECT ...
   WHERE ...
   EOF
   ```
3. **大结果集分页**：`LIMIT 50 OFFSET 100` 配合 `COUNT(*)` 估计总量。
4. **编码**：CSV 默认 UTF-8，shell 终端需支持中文（建议 `LANG=zh_CN.UTF-8`）。
5. **结果落盘**：复杂结果重定向到 `/tmp/result.csv` 便于 Agent 反复读取。

### 常见坑

- `plan_YYYY.first` 的值是"物理" / "历史"，用 `LIKE '%物%'` 匹配物理类。
- `plan_YYYY.second` 表述多样（"化学""化学或生物""不限"），用 `LIKE '%化%'` `LIKE '%生%'` `LIKE '%不限%'` 组合匹配。
- `score_phyYYYY` 与 `plan_YYYY` 用 `inst_code` + `major_code` 关联，**`inst` + `major` 文本不一致**时一定要用编码。
- `rank_phyYYYY.score` 类型是 TEXT，比较时用 `CAST(score AS INTEGER)` 避免字典序问题。
- 同一专业在不同年份可能换编码，跨年比较时要校验。
- `score_his2022` 不存在，分析历史类时仅能从 2023 起。

## 💬 提示词模板

完整使用流程与单点提问模板见 [README.md#-单点提问可选](./README.md#-单点提问可选)。

主工作流是 `templates/question.md` → `templates/report.md` 一次完成，单点提问模板只用于"只想问一个具体问题"的场景。

### 多轮收集信息（AI 主动追问）

如果考生只在对话里给了一部分信息（例如只说了分数，没说选科组合），**不要因为信息不全就拒绝生成方案，也不要让家长去翻 `question.md` 自己填**。AI 应主动按 [templates/question.md](../templates/question.md) 的字段逐项追问，每次问 1-3 个尚未明确的字段，凑齐后按 [templates/report.md](../templates/report.md) 输出志愿填报方案。

**追问顺序建议**（按"对方案影响从大到小"排）：

1. **必填项先行**：分数 → 高考年份 → 首选科目 → 选科组合
2. **强影响项**：想学专业 / 不想学的专业（直接决定志愿筛选）
3. **弱影响项**：升学或就业方向、学校位置、家庭就业资源（影响第 5 节院校介绍与最终志愿排序）
4. **可跳过项**：其他（如身体条件、特殊要求），明确说"如无则跳过"

**追问方式示例**：

> 为了给你生成完整志愿填报方案，我还需要补几个信息：
> 1. 选科组合是什么？（如 化学+生物 / 物理+化学 / 不限）
> 2. 想学的专业方向有哪些？（如 计算机、电子信息；不填则不限）
> 3. 升学或就业方向？（如 本科就业 / 读研 / 考公；不填则未定）

**关键规则**：
- **不要让家长一次性填完整张问卷**——分轮问，每次 1-3 项，降低填写门槛
- **缺省值不用问**——例如"其他"字段直接跳过
- **凑齐后立刻跑 `templates/report.md`**——不要再追加追问
- **回显时仍按 question.md 完整 9 项展示**（包括追问过程中明确说"用缺省值"的项）

## 📁 方案存档约定

AI 输出的方案应保存到本地 `reports/` 目录，**该目录已被 `.gitignore` 忽略，不会提交到仓库**。

### 文件命名

**用时间戳，不用考生姓名**——AI 通常不知道考生姓名，且时间戳排序更可靠。

格式：

```
reports/YYYYMMDD-HHMMSS.md
```

| 字段 | 说明 |
|---|---|
| `YYYYMMDD` | 生成日期，8 位数字 |
| `HHMMSS` | 生成时间，6 位数字（24 小时制） |

**示例**：

- `reports/20260626-143022.md` — 2026-06-26 14:30:22 生成的方案
- `reports/20260626-143022-v2.md` — 同日再次追问后的更新版
- `reports/20260628-091530.md` — 后续追问生成的新版本（独立文件）

**命名规则**：

- 同一天同一会话多次生成：后缀加 `-v2` / `-v3`
- 不同日期生成：独立文件（不覆盖前一次的）
- 不用考生姓名（隐私 + AI 通常不知道）
- 不用空格 / 中文 / 特殊字符（保证文件系统兼容）

### 目录结构

```
reports/
├── 20260626-143022.md                 # 第 1 份方案（志愿填报方案正文）
├── 20260626-143022-recommended.csv    # 志愿推荐表（默认 200 条：向上 100 + 向下 100，按代表位次升序）
├── 20260626-143022-full.csv           # 全部 ~1000 个候选评分大表
├── 20260626-143022-v2.md              # 同日追问更新版
├── 20260626-143022-v2-recommended.csv # 更新版对应推荐表
```

## 数据库分析经验

### 1. 数据库结构获取方法
- **优先读 [docs/db_schema.md](./docs/db_schema.md)**：表命名、字段、关联、坑点全部齐备，不要直接 `.schema` 表反推
- 仅在不确定当年 CSV 是否有新字段、或文档与数据库实际结果不一致时，才用 `sqlite3 data.sqlite3 ".schema table_name"` 兜底验证
- 使用 `sqlite3 data.sqlite3 ".tables"` 查看所有可用表
- 使用 `sqlite3 -header -column data.sqlite3 "SELECT * FROM table_name LIMIT 5;"` 查看表的前几行数据，了解数据格式

### 2. 分析流程经验
- **分步查询策略：** 先查招生计划，再查录取分数，最后查排名对应关系
- **数据验证步骤：** 通过 `COUNT(*)` 确认表有数据，通过 `LIMIT` 查看数据样本
- **模糊匹配技巧：** 使用 `LIKE '%关键词%'` 查找院校名称
- **多表关联方法：** 分别查询计划表、分数表、排名表，然后手动关联分析

### 3. 常用查询模式
```sql
-- 查看表结构和数据样本
SELECT * FROM table_name LIMIT 5;

-- 查找特定院校
SELECT DISTINCT inst FROM table_name WHERE inst LIKE '%南京%';

-- 查询招生计划（注意不同年份字段映射可能不同）
SELECT * FROM plan_YYYY WHERE inst = '南京大学' AND first = '物理';

-- 查询录取分数
SELECT * FROM score_phyYYYY WHERE inst = '南京大学';

-- 查询分数对应排名
SELECT score, rank FROM rank_phyYYYY WHERE score IN (分数列表);
```

### 4. 数据分析技巧
- **排序重要性：** 最终结果要按排名排序（排名数字从小到大）
- **对比分析：** 多年数据对比时要注意分数对应的排名变化
- **结果验证：** 通过多个角度验证数据合理性
- **异常检测：** 关注数据异常值和缺失值

### 5. 输出格式规范
- 表格形式展示，包含：专业、招生人数、最低录取分、省内排名
- 按排名排序，竞争激烈的专业排在前面
- 提供总结性分析，指出最难录取和相对容易的专业
- 对比多年数据变化趋势
- **方案必须回显 question.md 填写结果**：方案头先列出"考生信息（question.md 填写内容回显）"小节，把 [question.md](../templates/question.md) 里所有字段原样展示（分数、首选科目、选科组合、想学专业、不想学的专业、升学或就业方向、学校位置、家庭就业资源、其他），便于家长核对原始输入；缺省值也要明确标注，避免漏读为"没填"

### 6. 位次驱动的等效分换算（核心基石）

> **重要**：这是整个报考分析的基石。**等效分错了，下游一切结论都错**——风险判断、最终志愿表全部依赖准确的等效分换算。

#### 6.1 三步流程（必须严格按顺序执行）

| 步骤 | 输入 | 输出 | 数据来源 |
|---|---|---|---|
| 1. 今年分数 → 今年位次 | 考生原始分（如 500） | 考生位次（如 50000） | `rank_phy今年` 或 `rank_his今年` |
| 2. 今年位次 → 去年等效分 | 考生位次（50000） | 去年等效分（如 490） | `rank_phy去年` 或 `rank_his去年` |
| 3. 今年位次 → 前年等效分 | 考生位次（50000） | 前年等效分（如 510） | `rank_phy前年` 或 `rank_his前年` |

**关键点**：步骤 2、3 用的是同一个位次（考生今年的位次），不是去年的分数。位次就是沟通今年与去年的"坐标轴"。

#### 6.2 反查去年/前年的精确写法

去年、前年的 rank 表通常不会刚好命中"考生位次"这个整数（因为考生位次是今年分数对应的，不是去年某个精确分数对应的）。用"不大于考生位次的最大位次"近似：

```sql
SELECT score, rank FROM rank_phy去年
WHERE CAST(rank AS INTEGER) <= 考生位次
ORDER BY CAST(rank AS INTEGER) DESC LIMIT 1;
```

#### 6.3 CAST 的强制要求

- `rank_phyYYYY.score` 和 `rank` 都是 TEXT 类型
- 所有比较必须 `CAST(... AS INTEGER)`
- 正确：`WHERE CAST(score AS INTEGER) = 500`、`WHERE CAST(rank AS INTEGER) <= 50000`
- 错误：`WHERE score = 500`、`WHERE score >= 421`（TEXT 下 `5 > 421`）

#### 6.4 出方案前必做的校验

- [ ] 考生位次是否在 0 ~ 该年一分一段表最大值之间（异常值说明第 1 步有误）
- [ ] 反查去年、前年是否都返回了恰好 1 行（0 行或多行都是异常）
- [ ] 去年等效分、前年等效分与今年分的差值是否在 ±30 分以内（差值过大说明位次突变或题目难度异常）
- [ ] 历史类考生如要反查 2022：先确认 `score_his2022` 不存在，跳过前年、只到去年

#### 6.5 反例：为什么不能用直接分数对比

❌ "今年 500 分 = 去年 500 分，所以今年 500 分能上去年 500 分的学校"
- 题目难度不同、报考人数不同、招生计划数不同
- 同一个 500 分在不同年份对应的位次差异可能 1-2 万名

✅ 唯一正确的对比方式：用**今年位次**反查**去年分数**。位次是稳定的，分数是不可比的。

#### 6.6 完整参考 SQL（2026 年物理类 500 分考生示例）

```sql
-- 第 1 步：今年分数 → 今年位次
SELECT score, rank FROM rank_phy2026 WHERE CAST(score AS INTEGER) = 500;
-- 假设返回 rank = 50000

-- 第 2 步：用 50000 反查去年 rank 表（去年等效分）
SELECT score, rank FROM rank_phy2025
WHERE CAST(rank AS INTEGER) <= 50000
ORDER BY CAST(rank AS INTEGER) DESC LIMIT 1;
-- 假设返回 score = 490（去年等效分）

-- 第 3 步：用 50000 反查前年 rank 表（前年等效分）
SELECT score, rank FROM rank_phy2024
WHERE CAST(rank AS INTEGER) <= 50000
ORDER BY CAST(rank AS INTEGER) DESC LIMIT 1;
-- 假设返回 score = 510（前年等效分）
```

#### 6.7 失败处理

- **去年 rank 表缺失**：只展示今年 + 前年两行，标注"去年数据缺失"
- **反查返回 0 行**：位次异常，重新核对第 1 步结果
- **反查返回多行**：SQL 错误（`LIMIT 1` 之外不应有结果），检查 WHERE 条件
- **去年/前年与今年分差 > 30**：异常，标红提示人工复核（可能是疫情年、政策年等特殊情况）。注：此为分数异常的粗判，**严格上应看位次**（分数跨年不可比，位次跨年可比）
- **今年 score 表缺失**（典型场景：2026 年 6 月出方案时 `score_phy2026` / `score_his2026` 尚未发布）：**候选池建立流程不变**——4 张候选表本来就是从去年、前年 score 表提取（设计上就没用今年 score），JOIN 今年 plan 表取计划数。所以**方法上不降级**，只是今年 score 数据无法作为参考；提醒家长"今年录取分尚未发布，方案仅基于历史数据，等 7-8 月分数发布后建议复核"。定位分析节 + 志愿推荐表需明确标注"今年 score 表尚未发布"

#### 6.8 等效分锚定原则

**核心原则**：不能用"（去年等效分 + 前年等效分）/ 2"作为划线基准——平均后的分数在历史上从未存在过，对应的"录取格局"也无从查证。

**正确做法**：以考生今年的位次为坐标轴，分别用去年、前年的实际录取数据建立候选池，让大模型基于考生偏好做软筛选。

1. **位次锚定**：用考生今年位次反查去年 rank 表 → 得到去年等效分 `E_去年`；反查前年 rank 表 → 得到前年等效分 `E_前年`。
2. **候选池建立**：以 `E_去年` 和 `E_前年` 为锚点，SQL 提取上下各 N 个志愿（默认 N=300，可调整到 500）。
3. **合并去重**：4 张候选表 → 按 (院校, 专业) 去重 → 估计剩 800-1200 个志愿。
4. **大模型软筛选**：基于 question.md 软筛选（不硬编码偏好到 SQL），AI 凭常识对每个候选志愿评分（综合分 0-10），全部评分结果写入 `full.csv`。
5. **志愿推荐表选取**：以考生今年位次 `P` 为分界，**向上 100 条 + 向下 100 条 = 200 条**（默认）作为报告主表，按代表位次升序排列。详见下方 6.8.1 节。
6. ~~风险评估~~：**本节已删除**——用户决定不在报告里写风险章节，所有稳定性/计划数变化等定性判断由 AI 写进志愿推荐表的评语列。

> **重要**：本方案不再使用"冲 / 稳 / 保"档位概念。所有候选志愿按"**代表位次升序**"排列（位次小=门槛高=难录取=排前；位次大=门槛低=易录取=排后）——这是辽宁"分数优先、遵循志愿"投档逻辑决定的：考生按志愿表顺序检索，**第一个位次够的就录取**，所以代表位次小的志愿必须排前面（虽然今年分数够不上会跳过），不能把代表位次大的志愿排到前面。
>
> **代表位次** = min(去年位次, 前年位次)（两年都有数据时）。位次跨年可比，分数跨年不可比。

详见 [templates/report.md 第 2 节](./templates/report.md#2-志愿筛选按等效分锚定)。

##### 6.8.1 志愿推荐表选取规则（默认 200 条）

> **核心**：不以"综合分降序取 Top N"作为报告主表筛选条件——综合分高的志愿往往代表位次小（门槛高），全集中在冲段，违反辽宁"分数优先、遵循志愿"检索逻辑下"志愿表必须有命中区志愿"的基本要求。

**选取逻辑**：

```
考生今年位次 P（分界点）
   │
   ├── 向上 N 条：代表位次 < P 的候选中，按 AI 综合分降序取前 N
   │
   └── 向下 N 条：代表位次 ≥ P 的候选中，按 AI 综合分降序取前 N

合并 2N 条 → 按代表位次升序排列（即志愿填报顺序）
```

**关键提醒**：

- **"top N" 是按 AI 综合分（推荐分数）排序**，**不是按实际录取分数**
- 综合分反映"该志愿对考生的推荐价值"，与"该志愿的实际录取难度（代表位次）"是不同维度
- N 默认 = 100，推荐表 = **200 条**（覆盖完整梯度）
- 命名：不叫 "Top N"——"Top" 隐含"最优/最前列"，但 N 条是覆盖完整梯度的"推荐"，不是"客观最优"
- 报告主表正式名称：**志愿推荐表**（不是 "Top 200" / "Top 150" 等）

**N 调参建议**：

| 考生场景 | N 取值 | 推荐表数量 | 理由 |
|---|---|---|---|
| 600+ 高分段 | 80-100 | 160-200 | 高分志愿稀疏 |
| 500-600 主流分段 | 100 | 200 | **默认** |
| 450- 中低分段 | 100-120 | 200-240 | 低分段志愿密集 |
| 考生偏好宽（不限地区/专业） | 120-150 | 240-300 | 候选池大 |
| 考生偏好窄（指定专业+城市） | 60-80 | 120-160 | 候选池本身已小 |

**边界处理**：
- 某方向不足 N 条：取全部，不补（参考 6.9.4 降级原则）
- 某方向超过 N 条：按综合分降序取前 N
- 候选总池不足 2N 条：如实展示全部，不硬填

**CSV 文件命名**：
- 推荐表 CSV：`reports/YYYYMMDD-HHMMSS-recommended.csv`（语义准确"推荐志愿表"）
- 完整评分大表 CSV：`reports/YYYYMMDD-HHMMSS-full.csv`（全部 ~1000 条）
- 不再使用 `top150.csv` / `top200.csv` 等命名（避免与 Top N 概念混淆）

##### 6.8.2 决策记录：从"Top 150 综合分降序"到"志愿推荐表（向上 100 + 向下 100）"

> **重要**：本节是 2026-06-26 实操反思沉淀，**必读**。

**问题发现**：
首次按"综合分降序取前 150"生成报告后，发现 Top 150 中 **139/150** 个志愿的代表位次 < 12000，远低于考生位次 13066——**全集中在高分冲段**，命中区（12000-14000）仅 6 个，保段（> 14000）仅 5 个。这违反辽宁"分数优先、遵循志愿"检索逻辑下"志愿表必须有命中区志愿"的基本要求。

**为什么会出现这个问题**：
"综合分"反映"AI 对该志愿的推荐价值"，主要取决于学校层次 + 专业热度——这两者都倾向于高分学校（C9 985 王牌专业综合分最高）。所以"综合分降序取 Top N"本质上是在"高分学校中挑 AI 最推荐的"，与"覆盖完整志愿梯度"的目标错位。

**正确做法**：
以考生位次 P 为锚点，向上（< P）+ 向下（≥ P）分别取综合分 top N——
- **向上 N**：覆盖高分冲段（虽然够不上会跳过，但必须保留，作为"分数够高时从这开始命中"）
- **向下 N**：覆盖命中区 + 保段（最可能被录取的志愿集中在这里）
- **合并后按代表位次升序排列** → 这正是辽宁志愿表的填报顺序

**为什么不叫 "Top N"**：
- "Top N" 隐含"最优/最前列"——但 N 条是覆盖完整梯度的"推荐"，不是"客观最优"
- "志愿推荐表" 更准确：这是 AI 推荐给家长考虑的志愿集合
- N 默认 = 100 + 100 = 200（2N），可按考生情况调整 50-150

#### 6.9 志愿筛选流程（按等效分锚定）

本节介绍完整的志愿筛选流程：SQL 取数 → 合并去重 → 大模型软筛选 → 输出完整评分大表。

##### 6.9.1 候选池建立（SQL 阶段）

以等效分 `E` 为锚点，分别从去年、前年录取分表中提取候选志愿：

```sql
-- 去年 E 以上：录取分 ≥ E，按 min_score ASC 取前 N
SELECT s.inst, s.major, p.plan_num, s.min_score,
       (SELECT rank FROM rank_phy去年 WHERE score = s.min_score) AS 位次
FROM score_phy去年 s
JOIN plan_今年 p ON p.inst_code = s.inst_code AND p.major_code = s.major_code
WHERE p.first LIKE '%物%'  -- 或 '%历%'
  AND (p.second LIKE '%不限%' OR p.second LIKE '%化%' OR p.second LIKE '%生%')
  AND CAST(s.min_score AS INTEGER) >= E_去年
ORDER BY s.min_score ASC LIMIT N;

-- 去年 E 以下：录取分 < E，按 min_score DESC 取前 N
... WHERE CAST(s.min_score AS INTEGER) < E_去年 ORDER BY s.min_score DESC LIMIT N;
```

**N 默认 = 300**（不够可调整到 500）。

为什么 N=300：
- 4 张候选表共 1200 行原始数据，按 (院校, 专业) 去重后约 800-1000 个志愿
- 给大模型足够候选做软筛选（context 占 ~30-50K tokens），又不至于淹没细节
- N=300 是"候选丰富度"和"上下文成本"之间的经验折中值，不是数学最优解

**N=300 的分数跨度是位次不敏感的**（实测）：

| 锚点分数 | N=300 实际跨度 | 单分候选数 | 含义 |
|---|---|---|---|
| 600+（高分段） | 8 分（600-608） | 稀疏 | 每分人数少，300 行天然拉开 |
| 500（中分段） | 9 分（491-500 或 501-510） | 32 | 中等密度 |
| 450（中低分段） | 9 分（450-459） | 25 | 与中分段跨度相当，无明显差异 |

> **结论**：N=300 在各分数段都覆盖约 ±10 分，**"低分段挤在一分内"是错的**——所有分数段跨度差不多。

**调整建议**：

| 考生场景 | N 取值 | 理由 |
|---|---|---|
| 600+ 高分段 | 300 充裕 | 高分志愿本来就稀疏 |
| 500-600 主流分段 | 300 充裕 | 覆盖"接近 E 的核心段" |
| 450- 中低分段 | 300 充裕 | 跨度跟中分段一样，无需特殊调大 |
| 考生偏好宽（不限地区/专业） | 调到 500 | 扩大核心段覆盖 |
| 考生偏好窄（指定城市+专业） | 300 足够 | 候选池本身已小，无需扩大 |

> **决策记录：为什么不在 SQL 端硬筛"辽宁周边"**  
> 数据库 `plan_YYYY` 没有省份/城市字段，硬筛只能靠 `inst` 名称 LIKE 匹配城市/省份名。但这条路**边界模糊、维护成本高**，不可行：
> - **边界模糊**：很多本省院校名不直接带城市字样（如"抚顺石油学院"带"抚顺"算辽宁，但"辽宁石油化工大学"虽在抚顺却靠"辽宁"关键词命中）；"吉林大学珠海学院"会被 '%吉林%' 误匹配
> - **跨年不稳定**：新设院校、搬迁校区、改名（如"沈阳化工学院"→"沈阳化工大学"）都要同步更新硬筛列表
> - **改用大模型软筛**：question.md "学校位置"偏好在 6.9.3 大模型软筛阶段处理——大模型能根据院校所在城市常识判断"是否辽宁周边"，比硬编码更灵活

**前年同样**：以 `E_前年` 为锚点，输出 4 张候选表。

##### 6.9.2 合并去重

4 张候选表 → 按 (院校, 专业) 去重 → 估计剩 800-1200 个志愿。

**去重规则**：
- 同一 (院校, 专业) 在多张表出现 → 合并到一行，保留完整字段（去年分 + 前年分 + 今年计划数 + 双方位次）
- 只有一年有数据的志愿 → 另一年标 "—"
- 同一专业不同代码 / 不同方向（如同一个"辽宁大学 临床医学"多个方向）→ 保留多行

##### 6.9.3 AI 评分与排序（凭常识，不硬编码）

将 ~1000 个合并去重候选志愿交给 AI 处理。AI 任务分三步：

**1. AI 凭常识逐项打分**（不硬编码）

AI 拥有全国高校的常识知识——知道"辽宁大学在沈阳、211、应用经济学是双一流学科"——**不需要字典就能判断**。

**严禁硬编码**：
- ❌ 写 `CITY_SCORES = {"北京": 10, ...}` 这种大字典
- ❌ 写 `if "沈阳" in inst: score += 10` 这种大 if-else 链
- ❌ 把"辽宁周边"作为 SQL LIKE 条件硬筛（边界模糊、维护成本高，详见决策记录）
- ✅ 全部交给 AI 凭常识综合判断

**Python 脚本只做"无判断的机械工作"**：
- 合并去重（按 inst + major_code）
- 计算代表位次 = min(去年位次, 前年位次)
- 评分表格的格式化输出

**2. 评分原则由 AI 临时设计**（通用方法论，**不写死在 AGENTS.md**）

评分维度、权重、范围根据**本次考生情况**（分数、选科、偏好）临时确定，写入报告的"评分原则"章节。

> **AGENTS.md 不写具体考生的评分维度表**——那是 reports/ 报告的内容，不是通用规则。

设计评分维度时，AI 应考虑的方向（**通用指引，不是硬性清单**）：
- 学校层次（C9/985/211/双一流/一本/普通）
- 专业热度（基于考生偏好加权）
- 地理位置（基于考生位置偏好加权）
- 计划数稳定性
- 中外合作（学费高，通常扣分）
- 特殊行业（农林地矿等，通常扣分）
- 其他 AI 认为对当前考生有意义的维度

**3. 输出评分大表格**（写入报告）

每行简化格式：

```
| 序号 | rep | 院校 | 专业 | 计划数 | 前年分/位次 | 去年分/位次 | 综合分 | 评语 |
```

> **注意**：**不**单列"位次波动"——大表里前年分/位次与去年分/位次两列已经直接体现波动，再算一列是冗余。录取分稳定性由 AI 凭常识写进评语（例如"该校该专业连续两年录取分稳定"或"该中外合作专业去年位次大幅回落"）。

**不列出每个维度分数**（避免冗长，评分原则章节已说明）。

**4. 排序键**：

| 排序键 | 方向 | 理由 |
|---|---|---|
| 主键：代表位次 | **升序**（位次小→大） | 位次跨年可比；分数跨年不可比。位次小=门槛高=难录取=排前 |
| 同分 tie-breaker 1 | 计划数大优先 | 招生稳定 |
| 同分 tie-breaker 2 | 综合分高优先（AI 推荐度） | 相同代表位次下 AI 评分越高越优 |

**代表位次定义**：min(去年位次, 前年位次)（两年都有数据时）。

> **排序说明**：max(代表分) 降序 与 min(代表位次) 升序 **排序结果一致**（分与位次严格反相关），但**用位次更严谨**（跨年可比）。

**5. 两套输出：完整评分大表 + 志愿推荐表**

AI 评分完成后产出**两套 CSV 输出**（详见 [6.8.1 选取规则](#681-志愿推荐表选取规则默认-200-条)）：

- **完整评分大表（`full.csv`）**：全部 ~1000 个合并去重候选志愿的评分结果，**不压缩**——家长能直接看到的"原始信息"，AI 评分过程透明可查
- **志愿推荐表（`recommended.csv`，默认 200 条）**：以考生位次为分界，向上向下各取综合分 top 100，合并后按代表位次升序排列——写入报告主表

**不要**直接按"综合分降序取 Top N"作为报告主表筛选条件——会全集中在冲段，违反辽宁检索逻辑。详见 [6.8.2 决策记录](#682-决策记录从top-150-综合分降序到志愿推荐表向上-100--向下-100)。

如果软筛后候选数 < 2N：如实返回，不补。**不要硬填**。

##### 6.9.3.1 反投机取巧条款（AI 必须严格遵守）

**核心原则**：AI 必须**逐一查看**每一个 (院校, 专业) 候选志愿后**凭常识判断**打分，**严禁**任何形式的批量自动化。

**严禁行为**：

1. ❌ 写 `CITY_SCORES = {"北京": 10, ...}` 这种大字典
2. ❌ 写 `if "985" in inst: score += 8` 这种 if-else 链
3. ❌ 写 Python 脚本批量打分（即使逻辑是"凭常识"也禁止）
4. ❌ 用通用模板套所有志愿（如所有 211 都给 8 分）
5. ❌ 评语空泛（"好学校""符合偏好"等套话）
6. ❌ 评语复制粘贴（同一评语出现在多个志愿）

**评语质检清单**（AI 自查）：
- [ ] 评语含具体院校名或专业名
- [ ] 评语含可核实事实（学科评估 / 招生人数 / 招生章程条款等）
- [ ] 评语不是空泛的"好学校""符合偏好"等套话
- [ ] 评语长度 30-100 字（过短可能是套话，过长可能是凑数）
- [ ] 评语在该志愿上是唯一的（不是复制粘贴别的志愿）

**报告里必须明文写出 AI 评分承诺**（在 report.md 2.2 节或附录 A），让家长知道 AI 按什么标准打分、可以如何质疑。

##### 6.9.3.2 分批打分机制（推荐做法，可调整）

**为什么分批**：
- ~1000 个候选志愿一次性评分 context 压力大（~30-50K tokens）
- 注意力衰减：AI 对列表中后段志愿的注意力可能下降，评分质量不均
- 输出控制：一次性输出 1000 行 CSV 容易截断或格式错乱

**推荐做法**：
- **每批 ~100 条志愿**（可调整到 50-200 之间）
- ~1000 个候选 → **约 10 批**完成
- 每批独立打分，输出中间 CSV

**跨批次校准**（保证 10 批之间分数尺度一致）：

每批开始前，用 **2 个"标定志愿"**重打一遍：

1. **首批**：AI 自行设计 2 个标定志愿
   - 标定志愿 A：明显应该高分的（如清华计算机）
   - 标定志愿 B：明显应该低分的（如某偏远二本）
2. **后续每批**：先用同样的 2 个标定志愿快速重打
   - 校准通过（与首批结果接近，差距 ≤ 0.5 分）→ 开始本批打分
   - 校准偏差大 → 调整评分原则后再打本批

**报告里写明**：
- 评分原则节写明"本次评分分 N 批，每批 X 条志愿"
- 让家长看到 AI 实际做法，不是"凭一句话说逐一查看"

##### 6.9.4 边界值与降级

- 某年份录取分缺失：保留该年字段为 "—"
- 某张 SQL 表返回行数 < N：如实返回，不补
- **不要硬填**——筛不到就如实保留，比凑数更安全

##### 6.9.5 AI 评分流程（评分原则 + 评分结果写入报告）

**完整流程**：

1. **AI 临时设计评分原则**（基于考生情况）
   - 维度数量根据考生情况定（不一定每次都 12 个）
   - 范围灵活（0-10 / 0-5 / -3 等）
   - 加权原则由 AI 决定（基于 question.md 偏好）
2. **评分原则写入报告**——让家长知道 AI 按什么打分
3. **AI 凭常识逐项打分**——不写 Python 字典，凭常识判断
4. **输出全部候选志愿评分大表格**（写入 `full.csv`）——简化版 11 列（序号/代表位次/院校/专业/计划数/前年分/前年位次/去年分/去年位次/综合分/评语）
5. **志愿推荐表选取**（默认 200 条）——按 [6.8.1](#681-志愿推荐表选取规则默认-200-条) 规则从 `full.csv` 中提取，写入 `recommended.csv` 和报告主表
6. **统计与建议**（写入报告）——代表位次区间分布、推荐表覆盖梯度、关键发现

**评分原则章节示例**（**具体内容在 reports/ 报告里**——AGENTS.md 不写死）：

> 本节由 AI 根据本次考生情况临时设计。例如：考生物化生 + 父亲 IT → AI 设计"IT 类专业加权"+"辽宁本省加权"等。
>
> 评分维度、权重、范围都不写死在 AGENTS.md。

**关键原则**：
- AI 评分**不硬编码**——不写大字典、不写大 if-else 链
- Python 脚本**只做机械工作**（合并去重、计算代表位次、按位次切分推荐表）
- 评分原则 + 评分结果**都写入报告**
- **完整评分大表（`full.csv`）全部附加**——家长能直接看到的"原始信息"，AI 评分过程透明可查
- **志愿推荐表（默认 200 条）作为报告主表**——覆盖完整梯度的精华版本
- **AGENTS.md 是通用规则文档，不写具体考生的评分维度表**——具体内容在 reports/ 报告里
